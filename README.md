# core

A system for execution, collaboration, and rights enforcement  
without relying on trust — based on verifiability, traceability, and reproducibility.

---

## Philosophy (Summary)

- Trust is not required — verifiability is
- Facts must be provable
- Systems should make lies difficult, not rely on honesty
- Decisions must be based on records, evidence, and procedures

## 原文（Japanese - Authoritative）

以下は本プロジェクトの統括設計書であり、最上位の基準文書である。  
この内容が最優先の判断基準となる。

統括設計書 v0.1
題名: 理念・目標・目的・小目的(テスト)による総括仕様

==================================================
0. この文書の位置づけ
==================================================

この文書は、本計画全体を統括する最上位の基準文書である。
個別チャット、個別仕様、個別モジュールは、すべて本書の理念・目標・目的に従う。
本書に反する詳細設計は採用しない。

本書の役割は以下のとおりである。

1. 設計思想の固定
2. 全体目標の固定
3. 目的の階層分解
4. テスト可能な小目的への分割
5. 個別仕様書の評価基準の提供
6. 将来の改訂時における原点の保持

本書は「何を守るべきか」を定義する。
個別文書は「どう実装するか」を定義する。

==================================================
1. 理念
==================================================

1-1. 根本理念

人を信用しなくても、権利行使・実務遂行・協働が可能な構造を作る。

1-2. 中心思想

・信頼ではなく検証可能性を用いる
・証明できる事実を優先する
・嘘を防ぐのではなく、嘘が通りにくい構造を作る
・主観や権威ではなく、履歴・証拠・手順・整合性で判断する
・弱者でも参加できるよう、構造そのものを再設計する
・AIは支配者ではなく補助者である
・価値の源泉は「再現可能な手順」「検証可能な結果」「追跡可能な履歴」にある

1-3. 社会的意義

現行の制度や専門サービスは、知識格差・費用格差・時間格差によって、
本来権利を持つ当事者が十分に戦えない構造を生みやすい。
本計画は、その不均衡を少しでも減らし、
誰でも利用できる補助基盤・手続基盤・実行基盤を作ることを目指す。

1-4. 理念上の禁止事項

・「有名な人が言ったから正しい」を採用しない
・「AIがそう言ったから正しい」を採用しない
・ 履歴不明の更新を認めない
・ 検証不能なブラックボックス判断を中核に置かない
・ いきなり巨大で複雑な仕組みを完成品として作ろうとしない
・ 利用者に過度な信頼や高度な知識を要求しない

==================================================
2. 目標
==================================================

2-1. 最上位目標

検証可能性・履歴性・経済的設計を用いて、
誰でも参加できる非信頼型の実行・支援・協働基盤を構築する。

2-2. 実践的な目標

・誰が何をしたかを追跡できること
・どの手順で何が出たかを後から確認できること
・同じ仕様に基づく複数実行を比較できること
・AIを含む揺らぐ出力でも、評価基準に基づいて扱えること
・重い機能を後回しにしても、骨格だけで価値が出ること
・個別モジュールを交換・更新・追加できること
・問題発生時に、感情ではなく手順と証拠で切り分けられること

2-3. 目標達成の基本方針

・基盤を先、重機能を後
・完全性より再利用性
・理想の完全分散より、まず検証可能な運用
・一発の正解より、比較可能な試行の蓄積
・巨大統合より、小単位モジュールの連結

==================================================
3. 目的
==================================================

本計画の目標を実現するため、以下の主要目的を設定する。

--------------------------------------------------
目的1. 仕様の根を固定する
--------------------------------------------------

定義:
全ての設計・実行・更新の基準となる文書体系を固定する。

意味:
後から設計がぶれたり、個別判断が暴走したりしないよう、
まず「何が正本か」を固定する。

達成条件:
・統括文書が存在する
・下位文書の役割分担が決まっている
・改訂ルールが決まっている
・用語定義が存在する

--------------------------------------------------
目的2. モジュールの系譜を追跡可能にする
--------------------------------------------------

定義:
どのモジュールが、どの版から派生し、何が変わったかを追えるようにする。

意味:
不正・事故・バグ・改善を区別するための基盤。

達成条件:
・module_id がある
・version_id がある
・parent_version_id がある
・content_hash がある
・変更理由が残る

--------------------------------------------------
目的3. 実行事実を記録できるようにする
--------------------------------------------------

定義:
誰が、いつ、何を、どの仕様で、どう実行したかを残す。

意味:
評価・比較・紛争処理・再検討の土台。

達成条件:
・実行日時が記録される
・入力参照が記録される
・出力参照が記録される
・実行者識別子が記録される
・成否が記録される

--------------------------------------------------
目的4. AIを含む非決定的出力を扱えるようにする
--------------------------------------------------

定義:
完全一致しない出力でも、評価可能な設計を作る。

意味:
AIの揺らぎを前提として、結果の質と整合性を判定できるようにする。

達成条件:
・完全一致以外の評価法がある
・必須要素の充足確認ができる
・複数回試行や複数実行者比較ができる
・収束や傾向を見られる

--------------------------------------------------
目的5. 危険を限定して実行できるようにする
--------------------------------------------------

定義:
モジュールを実行しても、被害や権限逸脱が広がりにくい構造にする。

意味:
事故や悪意を前提にしても、被害範囲を狭める。

達成条件:
・実行境界がある
・権限が最小化される
・秘密情報を分離する
・検証前のものをそのまま信用しない

--------------------------------------------------
目的6. 価値が小さく積み上がる仕組みにする
--------------------------------------------------

定義:
大規模完成を待たず、小さな成功・小さな利用・小さな検証で価値が生まれるようにする。

意味:
最初から巨大構想で止まらず、骨格から実利を発生させる。

達成条件:
・モジュール単位で試せる
・個別に評価できる
・改善が差分で回る
・最低限の構造だけで利用可能

--------------------------------------------------
目的7. 将来的な経済・仲裁・ガバナンスに接続できるようにする
--------------------------------------------------

定義:
MVPでは重くしないが、将来の担保・紛争・評判・ガバナンスを接続可能にしておく。

意味:
後から成長できる骨格を最初から持つ。

達成条件:
・拡張余地がある
・役割分担が壊れない
・履歴が後付け拡張を支える
・モジュール更新が系譜管理できる

==================================================
4. 小目的(テスト)
==================================================

ここでは、各目的が達成されているかを判定するための
最小テスト単位を定義する。
これは設計・仕様・実装の全段階で使う。

--------------------------------------------------
小目的1. 統括文書の存在テスト
--------------------------------------------------

テスト内容:
・理念、目標、目的、小目的が一つの文書に定義されているか
・その文書が上位基準として扱われているか

合格条件:
・文書が存在する
・役割が明示されている
・下位仕様との関係が書かれている

--------------------------------------------------
小目的2. 用語固定テスト
--------------------------------------------------

テスト内容:
以下の最低用語が定義されているか

・module
・manifest
・version
・hash
・signature
・execution
・evaluation
・invariant
・log
・fact
・interpretation

合格条件:
・主要語にブレない定義がある

--------------------------------------------------
小目的3. モジュール識別テスト
--------------------------------------------------

テスト内容:
任意のモジュールについて、識別に必要な情報があるか

必要項目:
・module_id
・version_id
・content_hash

合格条件:
・別人が見てもモジュールを一意に区別できる

--------------------------------------------------
小目的4. 変更追跡テスト
--------------------------------------------------

テスト内容:
あるモジュールの旧版と新版の関係が追えるか

必要項目:
・parent_version_id
・change_reason

合格条件:
・変更の系譜が追跡可能である

--------------------------------------------------
小目的5. 入力仕様テスト
--------------------------------------------------

テスト内容:
モジュールが必要とする入力が定義されているか

確認項目:
・入力種別
・必須項目
・受け渡し方法
・参照形式

合格条件:
・第三者が入力を用意できる

--------------------------------------------------
小目的6. 出力仕様テスト
--------------------------------------------------

テスト内容:
モジュールの出力が何であるか定義されているか

確認項目:
・出力種別
・必須要素
・禁止逸脱
・失敗条件

合格条件:
・第三者が出力の妥当性を評価できる

--------------------------------------------------
小目的7. 実行記録テスト
--------------------------------------------------

テスト内容:
1回の実行に対して、最低限の事実ログが残るか

最低項目:
・実行日時
・実行対象
・入力参照
・出力参照
・実行者識別子
・成否

合格条件:
・何が起きたかを後から説明できる

--------------------------------------------------
小目的8. 署名検証テスト
--------------------------------------------------

テスト内容:
仕様書またはモジュールに付与された署名を検証できるか

合格条件:
・署名の有無を確認できる
・署名不正を検知できる

--------------------------------------------------
小目的9. ハッシュ整合性テスト
--------------------------------------------------

テスト内容:
入力・出力・仕様に対してハッシュが取れ、改ざん検知ができるか

合格条件:
・1文字変更で差異が検出される
・一致/不一致が明確に判定できる

--------------------------------------------------
小目的10. 非決定的評価テスト
--------------------------------------------------

テスト内容:
AIを含むモジュールに対し、完全一致以外の評価基準があるか

候補:
・必須要素充足
・類似度閾値
・ルーブリック評価
・多数収束

合格条件:
・「ブレたから不明」で終わらない

--------------------------------------------------
小目的11. 不変条件テスト
--------------------------------------------------

テスト内容:
出力が揺れても、絶対守るべき条件が定義されているか

例:
・特定要素を必ず含む
・危険な命令を出さない
・指定形式を守る

合格条件:
・許容されるブレと禁止される逸脱が区別される

--------------------------------------------------
小目的12. 実行境界テスト
--------------------------------------------------

テスト内容:
モジュール実行範囲が無制限になっていないか

確認項目:
・権限境界
・実行領域
・外部接続条件
・危険操作の制限

合格条件:
・事故時の被害範囲を説明できる

--------------------------------------------------
小目的13. 秘密分離テスト
--------------------------------------------------

テスト内容:
秘密鍵や秘密情報が、通常のモジュール実行と分離されているか

合格条件:
・鍵が平文で混在しない
・秘密情報を直接持ち回らない

--------------------------------------------------
小目的14. 比較可能性テスト
--------------------------------------------------

テスト内容:
複数実行結果を比較できるか

確認項目:
・同一仕様比較
・異版比較
・複数実行者比較
・評価結果比較

合格条件:
・傾向差分を見られる

--------------------------------------------------
小目的15. 最小価値発生テスト
--------------------------------------------------

テスト内容:
重い決済・重い仲裁・重いガバナンスがなくても、
1つ以上の実用価値が発生するか

合格条件:
・小さな利用が成立する
・運用試験が可能
・学習と改善が回る

--------------------------------------------------
小目的16. 拡張余地テスト
--------------------------------------------------

テスト内容:
将来、経済・評判・仲裁・分散化を足しても、
既存設計が壊れないか

合格条件:
・責務分離が維持される
・拡張点が明示されている

==================================================
8. 最終要約
==================================================

この計画は、
「まず理念を固定し、
次に目標を固定し、
次に目的を分解し、
最後に小目的をテストとして置く」
という順で組み立てる。

最初から巨大機構を完成させるのではない。
まずは、何を守るか、何を測るか、どう分解するかを固定する。

この文書の役割は、
全ての後続仕様が迷走しないための原点を保存することである。

一言で言えば、

『信頼ではなく、検証可能な骨格を先に固定するための総括設計書』

以上。
